# Architecture

dot-cursor installs five layers of automation into a project. Each layer handles what it's best at, and they share state through git and the filesystem.

## The Five Layers

```
┌──────────────┬───────────────┬──────────────┬──────────────┬──────────────┐
│   Cursor     │ Copilot CLI   │ Git Hooks    │ GitHub       │ Cron         │
│   (IDE)      │ (terminal)    │ (lifecycle)  │ Actions      │ (optional)   │
├──────────────┼───────────────┼──────────────┼──────────────┼──────────────┤
│ Interactive  │ Quick tasks   │ Per-commit   │ PR-gated     │ Local backup │
│ Agent chat   │ plan/autopilot│ Auto-logging │ Server-side  │              │
│ .cursor/     │ .github/      │ .git/hooks/  │ .github/     │ crontab      │
│ AGENTS.md    │ AGENTS.md     │ session-logs/│ workflows/   │              │
└──────────────┴───────────────┴──────────────┴──────────────┴──────────────┘
         Shared: session-logs/  ·  AGENTS.md  ·  git history
```

### Layer 1: Cursor IDE

Cursor is the interactive development environment. It reads `.cursor/rules/` for coding guidelines and `.cursor/modes.json` for custom agents. You interact with it through Agent chat (`Cmd+L`), inline edit (`Cmd+I`), and `@`-references to files, rules, and agents.

**Model strategy:**
- **Opus 4.6** — Architecture reviews, complex multi-file reasoning
- **Sonnet 4.6** — Daily driver, session management, code generation
- **Haiku 4.5** — Tab completions, inline suggestions

**Key difference from Claude Code:** Cursor's effective context window is 70–120K tokens (vs Claude Code's 200K). Start new conversations more frequently, use precise `@file` references, and keep rules concise.

### Layer 2: Copilot CLI

GitHub Copilot CLI runs in the terminal for quick tasks, CI/CD scripting, and one-offs. It reads `AGENTS.md` and `.github/copilot-instructions.md` for project context.

**When to use Copilot CLI instead of Cursor:**
- Quick shell commands: `ghcs "find all files modified in the last hour"`
- CI/CD pipeline generation
- Terminal-based workflows where opening the IDE is overkill

### Layer 3: Git Hooks

Git hooks are the lifecycle automation layer. They fire on every git operation regardless of which tool initiated it — Cursor, Copilot CLI, raw terminal, or CI.

**Installed hooks:**

| Hook | Fires When | What It Does |
|------|-----------|--------------|
| `prepare-commit-msg` | `git commit` | Pre-fills a conventional commit template with type/scope from diff analysis |
| `post-commit` | After commit | Logs to `session-logs/activity-*.md`, reminds about `@session-logger` at 5+ commits |
| `post-checkout` | Branch switch | Surfaces most recent handoff file with age and preview |
| `pre-push` | `git push` | Blocks WIP commits on main/master, warns on 10+ unpushed commits, detects large files |
| `post-merge` | `git pull`/merge | Alerts on dependency/config changes, reminds to update lockfiles |

**Why git hooks and not Cursor hooks?** Cursor has no native hook system. Git hooks fire for every tool in the stack, making them the universal lifecycle layer.

### Layer 4: GitHub Actions

Actions are the server-side complement to local hooks. They run reliably regardless of laptop state, are team-visible, and gate PRs.

**Installed workflows:**

| Workflow | Trigger | What It Does |
|----------|---------|--------------|
| `pr-description.yml` | PR opened | Parses conventional commits, groups by type, auto-fills PR body |
| `commit-lint.yml` | PR opened | Fails check if any commit doesn't match `type(scope): description` |
| `pr-review-checklist.yml` | PR opened | Flags: large PRs, missing tests, sensitive files, dep changes, AI rule changes |
| `session-maintenance.yml` | Daily + weekly | Archives stale logs, detects missing handoffs, generates weekly summary |
| `branch-hygiene.yml` | Weekly (Monday) | Detects merged/stale/drifted branches, opens cleanup issue |

**Why both hooks and Actions?** Hooks give immediate local feedback. Actions give server-side reliability, team visibility, and PR gating. They're complementary, not redundant.

### Layer 5: Cron (optional)

Local cron jobs provide a backup maintenance layer for session logs. Actions handle this reliably server-side, so cron is optional (belt + suspenders).

Install with: `bin/setup-hooks.sh --install-cron /path/to/project`

## Shared State

All five layers share state through three mechanisms:

### session-logs/

Git hooks write activity files. Cursor agents read them for context. Actions archive old ones and generate weekly summaries. Format:

```
session-logs/
├── activity-2026-03-02.md      # Auto-generated by post-commit hook
├── session-2026-03-02-1400.md  # Created by @session-logger agent
├── handoff-2026-03-02-1800.md  # Created by @handoff agent
├── weekly-2026-02-23.md        # Auto-generated by Actions weekly summary
└── archive/                    # Auto-archived by Actions session-maintenance
```

### AGENTS.md

A cross-tool instruction file in the project root. Read by Cursor, Copilot CLI, GitHub Copilot, Codex CLI, and Gemini CLI. Contains project-level AI instructions that apply regardless of tool.

### Git History

Conventional commits (enforced by `prepare-commit-msg` hook + `commit-lint` Action) create a structured log. The PR description workflow parses this into categorized changelogs. The weekly summary workflow extracts metrics from it.

## File Ownership

| File/Directory | Created By | Read By | Version Controlled? |
|----------------|-----------|---------|-------------------|
| `.cursor/rules/` | `migrate-to-cursor.sh` | Cursor | Yes |
| `.cursor/modes.json` | `migrate-to-cursor.sh` | Cursor | Yes |
| `AGENTS.md` | `migrate-to-cursor.sh` | Cursor, Copilot CLI, Codex | Yes |
| `.cursorignore` | `migrate-to-cursor.sh` | Cursor | Yes |
| `.github/workflows/` | `setup-actions.sh` | GitHub Actions | Yes |
| `.git/hooks/` | `setup-hooks.sh` | Git | **No** — run per clone |
| `session-logs/` | Git hooks, Cursor agents | All layers | Optional |
| `generated/` | `migrate-to-cursor.sh` | Human (manual paste) | No (gitignored) |

## Coexistence with dot-copilot

dot-cursor and dot-copilot install into non-overlapping directories within `.github/` and use separate config directories. Both read `AGENTS.md`. See the main README for the full compatibility matrix.

## Why Not Just One Tool?

Each tool has strengths the others lack:

| Capability | Cursor | Copilot CLI | Git Hooks | Actions |
|-----------|--------|-------------|-----------|---------|
| Interactive coding | ✅ | — | — | — |
| Terminal one-offs | — | ✅ | — | — |
| Per-commit automation | — | — | ✅ | — |
| PR gating | — | — | — | ✅ |
| Runs when laptop sleeps | — | — | — | ✅ |
| Team visibility | — | — | — | ✅ |
| IDE-agnostic | — | ✅ | ✅ | ✅ |
| Deep AI reasoning | ✅ (Opus) | — | — | — |

The five-layer approach covers all these cases without any single tool being overloaded.
